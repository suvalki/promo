
# 🎟️ PromoCode Manager

Микросервисное fullstack-приложение для управления промокодами с продвинутой аналитикой. Построено на паттерне CQRS с использованием NestJS, MongoDB, ClickHouse, Redis и BullMQ.

## 🌟 Возможности и фичи

- **Пользователи**: Регистрация, авторизация (JWT токены), просмотр профиля.
- **Промокоды**: Создание, редактирование, деактивация, установка лимитов (общие и на пользователя).
- **Заказы**: Гибкое создание заказов и применение к ним промокодов. Использование распределенных блокировок (Redlock) для защиты от Race Conditions (превышения лимита использований).
- **Аналитика**: Агрегация данных об использовании промокодов и тратах пользователей, предоставляемая мгновенно благодаря ClickHouse.
- **Глобальная фильтрация/Пагинация**: Поддержка server-side фильтрации, сортировки и пагинации для всех табличных данных.

## 🏗 Архитектурные решения

### 📊 ClickHouse: Таблицы и их назначение
В ClickHouse созданы следующие таблицы (движок `ReplacingMergeTree` для обеспечения уникальности `id` при обновлении):
- **User**: Денормализованные данные пользователей для быстрой фильтрации и поиска.
- **Promo**: Список промокодов с их лимитами.
- **Order**: Данные заказов, обогащенные информацией о пользователе и примененном промокоде (`userId`, `userName`, `promoCode` и т.д.).
- **PromoUsage**: История использований для аналитики "в реальном времени".

*Почему ClickHouse?* Столбцовая СУБД позволяет выполнять агрегационные запросы (например, "сумма трат всех пользователей по конкретному промокоду") значительно быстрее, чем классическая MongoDB, при этом не создавая нагрузку на основную транзакционную БД.

### 🔄 Синхронизация: MongoDB → ClickHouse
Синхронизация реализована через **Event-Driven** подход:
1. **Mongoose Plugin**: Каждая модель (`User`, `Order`, `Promo`, `PromoUsage`) подключена к `SyncPlugin`.
2. **Hooks**: При срабатывании хуков `post('save')` и `post('findOneAndUpdate')` ID документа отправляется в очередь **BullMQ**.
3. **Worker**: `SyncProcessor` извлекает задачу, запрашивает свежие данные из MongoDB (делая необходимые `populate`), обогащает их и выполняет `INSERT` в ClickHouse. 
   - Использование `ReplacingMergeTree` гарантирует, что при повторном `INSERT` с тем же `id` старая версия строки будет помечена на удаление, что заменяет `UPDATE`.

### 🔍 Server-side операции (Фильтрация, Сортировка, Пагинация)
Для обеспечения производительности на больших объемах данных все табличные операции вынесены на сторону БД:
- **ClickHouseQueryBuilder**: Кастомный класс для динамической сборки SQL-запросов.
- **Фильтрация**: Поддержка поиска через `ILIKE`, фильтрация по диапазонам дат (`dateFrom`/`dateTo`) и точным совпадениям (например, `userId`).
- **Сортировка**: Динамический `ORDER BY` на основе параметров `sortBy` и `sortOrder`.
- **Пагинация**: Реализована через `LIMIT` и `OFFSET`.
- **Безопасность**: Используются `query_params` (Parameterized Queries) для защиты от SQL-инъекций.

## 🚀 Быстрый старт

### Требования
- Установленный [Docker](https://www.docker.com/) и Docker Compose
- (Опционально) [Bun](https://bun.sh/) для локальной разработки

### Запуск всего приложения (Docker)
Поднять проект (Frontend + Backend API + Инфраструктура) можно одной командой:
```bash
docker-compose up -d --build
```

После запуска:
- **Frontend**: [http://localhost:5173](http://localhost:5173)
- **Backend API**: [http://localhost:3001](http://localhost:3001)
- **API Docs (Swagger)**: [http://localhost:3001/api/docs](http://localhost:3001/api/docs)

> *При первом запуске бэкенд автоматически инициализирует структуру таблиц в ClickHouse.*

### Локальная разработка (Альтернативный вариант)
Если вы хотите вносить изменения в код бэкенда с `hot-reload`:

1. Запустите только инфраструктуру:
   ```bash
   docker-compose up -d mongodb redis clickhouse
   ```
2. Установите зависимости и запустите приложение:
   ```bash
   bun install
   bun run start:dev
   ```
   *В этом режиме API будет доступно на порту `3000` (если не переопределено в `.env`).*

## 📊 Документация отдельных модулей

### Сервис Analytics
Модуль **Analytics** — это микросервис, отвечающий за сбор и агрегацию аналитических данных с использованием **ClickHouse** и кэширования через **Redis**.

- **Endpoint**: `GET /analytics/me` — Получает детальную статистику по текущему авторизованному пользователю.
- **Под капотом**: Выполняет сложные денормализованные выборки из ClickHouse с JOIN-ами агрегированных трат и кол-ва заказов. Использует `ClickHouseQueryBuilder` для 100% защиты от SQL инъекций.
- **Оптимизация**: Все ответные выборки кэшируются в экземпляре Redis на `120 000` мс для максимальной производительности при больших нагрузках.
